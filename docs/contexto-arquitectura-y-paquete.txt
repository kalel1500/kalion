Soy desarrollador de Laravel y hace un tiempo empecé a aprender a implementar la arquitectura limpia / hexagonal y el DDD dentro de proyectos Laravel.

Los conceptos que aprendí a implementar fueron:
- Separar el código en capas (Domain, Application, Infrastructure)
- Utilizar interfaces para definir contratos entre capas
- Implementar casos de uso para encapsular la lógica de negocio
- Utilizar repositorios para acceder a los datos
- Aplicar principios SOLID para mantener el código limpio y mantenible

Ahora al organizar mis proyectos siempre creo las carpetas Application, Domain e Infrastructure dentro de "src" para dejar la carpeta "app" de Laravel con los archivos mínimos (el AppServiceProvider que viene por defecto en la instalación, ya que incluso los que se crean con después con los comandos meto en "src").

Pero no solo tengo esas 3 carpetas directamente en "src" sino que además las separo por contextos para estructurar mejor el código. Al final siempre tengo una carpeta "Shared" para las clases más transversales.

Como ves lo que es común en toda la aplicación (como modelos y repositorios) lo pongo en Shared y lo que es especifico de cada contexto lo pongo dentro de su carpeta correspondiente. De esta forma tengo una estructura de carpetas bastante ordenada y fácil de entender.

Siguiendo esta estructura dejo la lógica de negocio dentro del dominio (en las validaciones de los VOs y los DTOs), los casos de uso concretos dentro de la capa de aplicación y lo que es parte de Laravel dentro de infraestructura (como los controladores, modelos y repositorios).

Al final en "app" siempre me creo un "DependencyServiceProvider" donde doy de alta los "singletons" de la aplicación para que Laravel sepa como resolver las dependencias de cada clase (como los repositorios). De esta forma, por ejemplo, si en un caso de uso necesito el repositorio de posts, lo que hago es inyectar la interfaz del repositorio y Laravel se encargará de resolver la implementación concreta que tengo en infraestructura (en este caso el EloquentPostRepository).


De esta forma un controller me puede quedar asi de limpio:
    final class PostController extends Controller
    {
        public function __construct(
            public readonly GetViewDataPostListUseCase   $getViewDataPostListUseCase,
            public readonly GetViewDataPostDetailUseCase $getViewDataPostDetailUseCase,
        )
        {
        }

        public function list(Request $request): View
        {
            $data = $this->getViewDataPostListUseCase->__invoke($request->input('tag'));
            return view('pages.post.list', compact('data'));
        }

        public function detail(string $slug): View
        {
            $post = $this->getViewDataPostDetailUseCase->__invoke($slug);
            return view('pages.post.detail', compact('post'));
        }
    }

Una de las cosas que más me gusta de esta arquitectura es "desacoplarme" de eloquent y que muera en el repositorio, asi lo uso para hacer consultas fácilmente, pero no lo devuelvo a ningún sitio para que en nigun caso se haga un bucle que acceda a una relación y me haga "n" querys (que es algo que me ha pasado varias veces y tengo una app antigua donde la home tiene más de 40 querys aunque use el eager loading que no siempre soluciona las cosas en apps grandes y es complicado acordarse siempre). De esta forma, en el repositorio hago la consulta con eloquent, pero luego transformo el resultado a una entidad del dominio (en este caso a un PostEntity) y devuelvo esa entidad al caso de uso. De esta forma tengo el control total sobre lo que se devuelve y evito problemas de rendimiento.

Este ejemplo es relativamente fácil, pero a la minima que una entidad tenga muchas relaciones y esas relaciones tengas más relaciones y si tengo que hacer esto en cada return (porque cada query necesitara ciertas relaciones) se me hace un mundo y tengo mucho código repetido. Además de que he tenido que crear un Value Object por cada campo de cada entidad cuando la mayoría son iguales (PostIdVo y CommentIdVo o PostContentVo y CommentContentVo)

Con este sistema el ejemplo de antes se puede reducir a esto:

    class EloquentPostRepository implements PostRepository
    {
        private string $model;

        public function __construct()
        {
            $this->model = Post::class;
        }

        public function all(): PostCollection
        {
            $data = $this->model::query()->with('comments')->get();
            return PostCollection::fromArray($data->toArray(), ['comments']);
        }

        public function find(IdVo $id): PostEntity
        {
            $data = $this->model::query()->with('comments')->findOrFail($id);
            return PostEntity::fromArray($data->toArray(), ['comments']);
        }

    }

    #[CollectionOf(PostEntity::class)]
    class PostCollection extends AbstractCollectionEntity
    {
    }

    class PostEntity extends AbstractEntity
    {
        public function __construct(
            public readonly IdVo|IdNullVo   $id,
            public readonly StringVo        $title,
            public readonly StringVo        $content,
            public readonly StringVo        $slug,
            public readonly IdVo            $user_id,
            public readonly TimestampNullVo $created_at,
        )
        {
        }

        #[RelationOf(UserEntity::class)]
        public function user(): ?UserEntity
        {
            return $this->getRelation();
        }

        #[RelationOf(CommentCollection::class)]
        public function comments(): CommentCollection
        {
            return $this->getRelation();
        }

        #[RelationOf(TagCollection::class)]
        public function tags(): TagCollection
        {
            return $this->getRelation();
        }
    }


Como ves al extender la entidad y la colección de las clases abstractas se pueden instanciar automáticamente con el "fromArray" y se pueden añadir las relaciones creadas en la entidad siempre que vengan en los datos de la query.
